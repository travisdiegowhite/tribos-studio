## Executive summary

The Phase A audit confirms the two production symptoms surfaced by
P1.4 (elevation = 0 on every Stadia-sourced route; Supabase 400/404s
on `assembleRouteContext` queries) and identifies one additional
correctness bug (RouterClient cache key omits context-affecting
fields, so cache hits can serve incorrect routes when training goal
or preferences change).

In total: **2 P0 findings, 2 P1 findings, 3 P2 findings.** Phase A is
sufficient; Phase B is not triggered.

P1.5 (beta release) **should not open** until F-1 (elevation
enrichment) and F-2 (RouteContext schema) are resolved. Both have
small, well-defined fix scopes (each ≈ a half-day) — total beta-gate
overhead is on the order of one day of focused work plus verification.

The new executor pipeline is structurally sound: registry, fallback
chain, telemetry, distance/duration extraction, and snapshot
assembly all preserve legacy behavior accurately. The bugs are
concentrated at two specific seams — the elevation field name and
the RouteContext Supabase queries — not in the architecture.

## Findings

### F-1 — Elevation is 0 on every Stadia-sourced and Mapbox-sourced route

- **Severity:** P0
- **Location:**
  - `src/routing/RouterClient/providers/StadiaProvider.ts:163, 233` (extraction)
  - `src/routing/RouterClient/providers/MapboxProvider.ts:141, 215` (hardcoded 0)
  - `src/utils/stadiaMapsRouter.js:384–398` (the legacy module's return shape, which never includes `elevationGain`)
  - `src/features/route-builder-v2/adapters/executorAdapter.ts` (no enrichment hook)
  - `src/hooks/route-builder/useAIGeneration.ts:128` (writes 0 to store)
  - `src/hooks/route-builder/useRouteEditing.ts:97` (writes 0 to store)
- **Symptom:** Every Route Builder 2.0 route generated via Stadia (road, commute) or Mapbox (any profile as fallback) has `elevation_gain_m: 0` in the store. Downstream, `increase_climbing` and `reduce_climbing` mutations in `src/routing/executor/ConstraintBuilder/handlers/increase_climbing.ts:39–45` reject every Stadia route with `"Region has insufficient elevation (0m per 30km) to honor increase_climbing."` This is the P1.4-observed failure.
- **Root cause:**
  1. `stadiaMapsRouter.js` does not extract elevation from Valhalla's response. Its return shape (lines 384–398) includes `coordinates, distance_m, duration_s, maneuvers, trafficScore, quietnessScore, roadClassification` — no `elevationGain` field.
  2. `StadiaProvider.solve` reads `result.elevationGain ?? 0`, so the `?? 0` branch is the only one that ever fires.
  3. The legacy production pipeline solves this by calling `getElevationData()` (Open-Elevation/OpenTopoData) or `fetchElevationProfile()` (Mapbox elevation) **separately** after the router returns, in `aiRouteGenerator.js` (10+ call sites) and `useRouteManipulation.js`.
  4. The new RB2 pipeline (`useAIGeneration` → `executorAdapter` → `Executor` → `RouterClient`) has **no equivalent enrichment step**. The snapshot flows straight from `buildSnapshot` to `setRouteStats` with `elevation_gain_m: 0`.
  5. Mapbox has the same shape of bug — Mapbox doesn't return elevation natively, legacy enriches separately, new doesn't.
  6. BRouter is unaffected — its response includes `properties['filtered ascend']`, which both pipelines extract correctly.
- **Evidence:** `docs/audit/t2.6/legacy-vs-new-extraction.md` "Smoking gun" section.
- **Fix scope:** Medium. Two options:
  1. **(Preferred)** Add an elevation-enrichment step in the executor adapter or in a new `RouteAnalysis` post-processor, called after `executor.generate()` / `executor.applyMutation()` returns. Reuses the existing `getElevationData()` from `src/utils/elevation.js`. Roughly: 50 LOC + tests in one new module + 4 call sites updated in `useAIGeneration` and `useRouteEditing`.
  2. **(Avoid)** Pull Valhalla's per-leg `elevation` arrays into `StadiaProvider`. Higher engineering cost (Valhalla returns elevation as encoded delta arrays per edge; reassembly is non-trivial) and doesn't help Mapbox.
- **Fix spec:** **T2.6.1 — Add elevation enrichment to executor result path.**

### F-2 — `assembleRouteContext` queries 2 non-existent objects and 1 non-existent column

- **Severity:** P0
- **Location:** `src/features/route-builder-v2/adapters/assembleRouteContext.ts:128, 188, 220`
- **Symptom:** Three Supabase queries fail (404 / 400 / 400). Every failure is silently swallowed; the assembler returns a degenerate context with empty preferences, no DB-sourced training goal, no recent rides, default Boulder start coordinate, and default 25 km/h speed assumption.
- **Root cause:**
  1. `user_preferences_complete` view (line 188) — no `CREATE VIEW` exists in any migration. 404.
  2. `training_context` table (line 220) — does not exist as a table; only `cafe_discussions.training_context` JSONB column. 400. Additionally, the columns it queries (`primary_goal`, `typical_ride_time`) live on `user_profiles` (migration 062) and don't exist anywhere else, respectively.
  3. `activities.polyline` (line 128) — column does not exist; the actual column is `map_summary_polyline` (migration 001). 400.
  4. The `try { ... } catch {}` pattern around each query (lines 184–209, 217–236, 127–141) treats failures as "missing data" rather than "schema bug", so nothing surfaces.
  5. `assembleRouteContext` does NOT throw `RouteContextError { kind: 'context_missing' }` for any of these. The only error it throws is `no_user`. Doc 2b §9.1 (line 744) says required-field failures should produce a synthetic clarify; the assembler does not honor that.
- **Downstream impact:**
  - StadiaProvider's `buildLegacyPreferences` receives an empty base object → no traffic-tolerance layering, no `avoidHills`, no `avoidTraffic` applied → all routes use the bare profile costing regardless of user setting.
  - Every user's routes generate from Boulder (`DEFAULT_START_COORD`).
  - `like_ride_id` mutations always fall through to the radial-loop seed because `recent_rides` is always `[]`.
  - The executor never receives a `context_missing` signal; the conversational layer (Phase 2) won't be able to drive clarification UX on missing context.
- **Evidence:** `docs/audit/t2.6/routecontext-schema-mismatches.md`.
- **Travis verification needed:** run the three `information_schema` queries listed in §"What Travis needs to confirm in Supabase" of `routecontext-schema-mismatches.md` to confirm the audit's reading.
- **Fix scope:** Medium. The schema fix is not a one-line column rename — the assembler is querying for fields (`home_longitude`, `home_latitude`, `typical_ride_time`) that have no source-of-truth column today. Needs an explicit decision on source-of-truth for each `FullRouteContext` field. Likely:
  1. Replace `user_preferences_complete` query with direct `user_profiles` query, with explicit column list.
  2. Replace `training_context` query with `user_profiles.primary_goal` read.
  3. Replace `activities.polyline` with `activities.map_summary_polyline` (and decode it if waypoints are actually needed; today they're left empty).
  4. Define how `start_coord` is sourced — either add a `home_lng/home_lat` column to `user_profiles` (small migration) or derive from most-recent activity start point.
  5. Make the assembler throw `RouteContextError { kind: 'context_missing', required_field: ... }` for fields the conversational layer treats as required.
- **Fix spec:** **T2.6.2 — Rewrite RouteContext Supabase queries against real schema.**

### F-3 — RouterClient cache key omits context — same-waypoint regenerations serve stale routes after preference change

- **Severity:** P1
- **Location:** `src/routing/RouterClient/cache.ts:138–161` (`cacheKeyForConstraint`)
- **Symptom:** The cache key is computed from `RouteConstraint` only (waypoints, profile, shape, distance/elevation targets, segment lists). It does **not** include any `RouteContext` field. But StadiaProvider's costing depends on `context.training_goal`, `context.user_speed_kph`, and `context.preferences` (which transitively control traffic tolerance, avoid_hills, avoid_traffic). A user who generates a route with goal=recovery, then changes goal to intervals, then generates again with the same waypoints (within the 5-minute TTL), will get the cached recovery-tuned route — not a re-computed intervals route.
- **Root cause:** Cache key was designed before the context-driven costing layer was wired in. The spec (`docs/router-client.md` §"Cache behavior") documents the constraint-only key as deliberate, but doesn't acknowledge that this collides with context-driven costing.
- **Evidence:** `cache.ts:138–161` — no `context` parameter passed into `cacheKeyForConstraint`. `RouterClient.ts:108` calls `cacheKeyForConstraint(normalised)` with no context input. StadiaProvider's `buildLegacyPreferences` clearly varies output by context fields.
- **Fix scope:** Small. Include `training_goal`, `user_speed_kph`, and a stable serialization of `preferences` (or a hash thereof) in the cache key. Roughly 20 LOC change.
- **Fix spec:** **T2.6.3 — Include context fields in RouterClient cache key.**

### F-4 — `assembleRouteContext` swallows all Supabase failures silently; spec contract is not enforced

- **Severity:** P1 (escalates to P0 once the conversational pipeline lands)
- **Location:** `src/features/route-builder-v2/adapters/assembleRouteContext.ts:184–209` (and identical pattern at lines 111–142, 217–236)
- **Symptom:** Doc 2b §9.1 (line 744) explicitly contracts: "if any required field is missing (e.g., user has no profile), the dispatcher emits a synthetic `clarify` rather than calling the executor with incomplete context." The current assembler throws `RouteContextError` only for `no_user`; every other failure (schema mismatch, RLS deny, network error) is caught and ignored. The executor receives a "valid" context populated with defaults and proceeds to generate routes that ignore the user's actual configuration.
- **Root cause:** The `try { ... } catch { /* return empty */ }` pattern was likely written defensively to handle "user is new and has no profile yet" — which is the right behavior in that case. But the same pattern absorbs real bugs (schema mismatches) and real errors (RLS deny), and the assembler doesn't distinguish "empty because missing" from "empty because broken".
- **Why P1 not P0 today:** The new pipeline currently fronts a stub conversational layer (`heuristicTranslation.ts`). Once Doc 2b ships, the dispatcher relies on `context_missing` to drive clarification, and silent-failure becomes user-facing wrong behavior.
- **Evidence:** Compare `assembleRouteContext.ts:184–209` to the spec's required-field contract at `docs/doc-2b-conversational-pipeline-spec.md:744`.
- **Fix scope:** Small once F-2 is in flight — split "field is null because user hasn't set it" (return empty) from "query failed unexpectedly" (throw `RouteContextError`). Likely folds into T2.6.2.
- **Fix spec:** Folded into **T2.6.2**.

### F-5 — `RouteSnapshot.elevations_m` (per-point elevation array) is never populated

- **Severity:** P2 (informational; same root cause as F-1)
- **Location:** `src/routing/executor/types.ts:340` (declaration); all three providers (no population).
- **Symptom:** `RouteSnapshot.elevations_m?: number[]` exists in the type but no provider ever populates it. Handlers that need per-point elevation (`elevationGainInScope_m` at `src/routing/executor/ConstraintBuilder/shared/elevationUtils.ts:38`) fall back to the totals, which are 0 for Stadia (F-1). Scoped `increase_climbing` mutations therefore have no per-point data to compute scope-restricted gain.
- **Fix scope:** Fold into F-1's fix. When elevation enrichment runs, it produces a per-point series; write it to `elevations_m`.
- **Fix spec:** Folded into **T2.6.1**.

### F-6 — Stadia maneuver / road-classification data dropped at adapter boundary

- **Severity:** P2 (acknowledged in `router-client.md` known follow-up #2)
- **Location:** `src/routing/RouterClient/providers/StadiaProvider.ts` — never reads `result.maneuvers`, `result.trafficScore`, `result.roadClassification`.
- **Symptom:** `change_climb_character` (which needs to know if existing climbs are punchy/sustained/rolling), `avoid_segment_by_property` (which needs to know road class), and future arterial-detection mutations have no data to work with. Today this is benign — those mutations either don't use the data or are stub implementations — but the data is being dropped at the seam.
- **Fix scope:** Small. Pass through into `RouteSnapshot.metadata.diagnostics` (a slot the executor spec contemplates but hasn't formalized). Defer until a consumer needs it.
- **Fix spec:** **T2.6.4 — (post-beta) Surface Stadia maneuver/classification data in RouteSnapshot.metadata.**

### F-7 — PostHog events arrive with a doubled `route_builder_routerclient_*` prefix

- **Severity:** P2 (acknowledged in `router-client.md` known follow-up #3)
- **Location:** `src/routing/RouterClient/RouterClient.ts:57–67` — `track()` calls `trackRouteBuilder(`routerclient_${event}`, ...)`, and `trackRouteBuilder` adds its own `route_builder_` prefix.
- **Symptom:** Events arrive in PostHog as `route_builder_routerclient_solve_called` rather than `routerclient_solve_called` as the spec documents.
- **Fix scope:** Small. Either extend `trackRouteBuilder` to accept a configurable prefix or call PostHog directly with the desired event name.
- **Fix spec:** **T2.6.5 — (post-beta) Clean up routerclient telemetry prefix.**

## Recommended fix sequence

1. **T2.6.1 — Add elevation enrichment to executor result path.** Unblocks chat-driven climbing mutations and gets correct elevation onto every route. Includes populating `RouteSnapshot.elevations_m` (F-5). **(P0)**
2. **T2.6.2 — Rewrite RouteContext Supabase queries against real schema, and surface failures as `RouteContextError`.** Restores preference-aware routing and honors the Doc 2b contract (F-4). Depends on Travis confirming schema via the `information_schema` queries in `routecontext-schema-mismatches.md`. Likely requires a small migration if `start_coord` source-of-truth lands as a new column. **(P0 + bundled P1)**
3. **T2.6.3 — Include context fields in RouterClient cache key.** Small, low-risk; ship alongside or after T2.6.2 once context fields are actually populated (otherwise the fix is a no-op). **(P1)**
4. **T2.6.4 — Surface Stadia maneuver/classification data in RouteSnapshot.metadata.** Post-beta. Needed when a consumer (Phase 2 mutation handlers) starts using the data. **(P2)**
5. **T2.6.5 — Clean up routerclient telemetry prefix.** Post-beta cosmetic. Worth doing during the next telemetry sweep, not earlier. **(P2)**

T2.6.1 and T2.6.2 are independent and can be parallelized. T2.6.3 has
a soft dependency on T2.6.2 (no value without populated context).

## Out-of-scope observations

These are not bugs in T2.x and not actionable in T2.6 follow-ups, but
worth tracking:

- The new `useRouteEditing.snapshotCurrentRoute` (`src/hooks/route-builder/useRouteEditing.ts:71–89`) and `setRouteStats` writes silently drop `elevation_loss_m`. The Zustand store's `routeStats` schema (`src/stores/routeBuilderStore.js:25`) is `{ distance_km, elevation_gain_m, duration_s }` only. This is consistent with the store but means "elevation loss" can never round-trip through the editor. Not a regression — the legacy store had the same shape — but a quiet data loss now that providers produce `elevation_loss_m`.
- The new pipeline (RB2) has no equivalent to the legacy `aiRouteGenerator.js` (≈2900 LOC), which is the production path for AI route generation. T2.x is replacing that piece by piece; the elevation enrichment gap (F-1) is one of the bigger pieces that hasn't been ported yet. There are likely several more "missing enrichment" steps that haven't been audited (POI annotation, infrastructure scoring, surface type detection, etc.). Calling these out is beyond Phase A scope, but the cutover plan should explicitly enumerate them before P1.5 broadens beyond the initial cohort.
- The runner script (`scripts/audit/t2.6/comparison-runner.ts`) should be preserved beyond T2.6 as `scripts/parity-check.ts` — a quick way to side-by-side the two pipelines anytime the new one changes meaningfully. Renaming and keeping it is recommended.
- The legacy `smartCyclingRouter.js` enriches its own results with `source: 'stadia_maps' | 'brouter_gravel' | 'brouter' | 'mapbox_fallback'` and `confidence: 1.0 | 0.9 | 0.8 | 0.7` decorators. The new pipeline reports only `metadata.provider_used: ProviderName` and drops the gravel/non-gravel BRouter distinction. The "follow-up #1" note in `router-client.md` proposes `metadata.provider_profile` for this; that should ship around the same time as the maneuver-data follow-up (T2.6.4).

## Methodology notes

### What this audit verified

- Side-by-side comparison of all three provider adapters against their legacy modules, line-numbered.
- Every Supabase query in `assembleRouteContext.ts` cross-referenced against `database/migrations/*.sql`.
- Fallback chain ordering in `registry.ts` against the hardcoded order in `smartCyclingRouter.js`.
- Telemetry event surface against the catalog in `router-client.md`.
- Hook data flow from `useAIGeneration` / `useRouteEditing` through the adapter, executor, and provider, looking for fields silently dropped or never populated.

### What this audit did NOT verify

- **Real-API runs.** The audit environment had no provider API keys. The runner script is ready (`scripts/audit/t2.6/comparison-runner.ts`); Travis to execute. Findings are derived from static analysis, which is sufficient for the elevation bug (the field name simply doesn't exist in the legacy response). Distance and geometry parity should be re-confirmed by the runs.
- **Visual geojson.io inspection** (Step 8 in the spec). Depends on runs.
- **Phase B**: ConstraintBuilder mutation handlers, MutationHandlers composition, ManualHandlers, Executor facade internals, executor adapter translation correctness for `applyMutations`, P1.2 hooks state propagation under mutation chains.

### Decision: stop at Phase A

Per the spec's bias toward stopping at Phase A unless findings are
systemic, Phase B is not triggered. Rationale:

- All findings are **localized to specific seams** — the StadiaProvider elevation read, the assembler Supabase queries, the cache key. None point to architectural problems with the executor design.
- The fallback chain, telemetry surface, distance/duration extraction, snapshot assembly, and provider adapter contracts all match legacy exactly or are deliberate improvements (`ProviderFailure` types, `supports()` filtering, request dedup).
- BRouter — which has the most provider-side complexity — passes audit cleanly.
- The two P0 bugs are exactly the symptoms P1.4 already observed. The audit confirms their root causes and gives them fix scope.

If T2.6.1 and T2.6.2 fixes uncover deeper issues — e.g., if elevation
enrichment exposes that the executor adapter doesn't re-emit modified
snapshots correctly — Phase B can be triggered as a follow-up audit.

### Recommendation for ongoing parity verification

- Promote `scripts/audit/t2.6/comparison-runner.ts` to `scripts/parity-check.ts` (renamed and kept).
- Add it to the route-builder dev workflow as an on-demand sanity check before any meaningful change to RouterClient or the providers.
- Don't run it in CI — it costs provider quota and the comparisons take 30–60s. A weekly manual run during the Phase 2 buildout is enough.
