# T1.1 — Distance Unit Contract Audit & Fix

**Status:** Ready for handoff
**Priority:** Critical — blocks T2 (executor rebuild)
**Estimate:** 1–2 days
**Depends on:** Nothing
**Blocks:** T1.2 (concurrent OK), T2.x (all executor work)

---

## Goal

Eliminate the silent km/m unit-mismatch bug class from the Route Builder codebase. After this task, every variable, function parameter, return value, type definition, and storage field that holds a distance value must be annotated with either a `_km` or `_m` suffix, and the values must actually be in the named unit.

This is the precondition for the new executor architecture, which mandates `_km` everywhere internally with conversion only at router boundaries.

---

## Background

Per the Route Builder architectural review (October 2024):

> `routeBuilderStore.js:19–22` documents distance as **km**, but `useRouteManipulation.snapToRoads` returns **meters**. Recipe for silent 1000× errors:
>
> ```js
> const { distance: distMeters } = await getSmartCyclingRoute(...);
> setRouteStats({ distance: distMeters }); // stored as km → 45,300 km
> ```

This is the canonical bug. There may be others. The task is to find them all.

---

## Scope

### In scope

- All TypeScript/JavaScript files under `src/` that touch distance values
- All Supabase schema columns storing distances
- All Zustand store fields storing distances
- All Edge Function code that processes distance values
- All `useRouteManipulation`, `useRouteOperations`, routing utility modules

### Out of scope

- Distance values inside third-party library calls (Mapbox, BRouter, Stadia, Open-Elevation responses) — those are converted at the boundary
- UI display strings (e.g., "32.5 mi" rendered for user) — those are formatted at presentation
- Other unit categories (time, elevation, power) — separate audit if needed

---

## Definitions

| Term | Meaning |
|---|---|
| Distance variable | Any variable, field, parameter, or return value whose numeric value represents a length (route distance, segment length, snap radius, etc.) |
| `_km` suffix | Variable name ends in `_km` or `Km` or `KM` (snake_case, camelCase, or constant) |
| `_m` suffix | Variable name ends in `_m`, `_meters`, or `Meters` |
| Boundary | Code site where a value enters from or exits to a third-party API |

## Step-by-step process

### Step 1: Discovery (no code changes yet)

Produce an audit report listing every distance variable in the codebase. Use `rg` (ripgrep) to find candidates:

```bash
# Find likely distance variables
rg -i "distance|length|radius" --type ts --type tsx --type js --type jsx -l src/
rg -i "(km|kilometer|meter|miles?)\b" --type ts --type tsx --type js --type jsx -l src/

# Find places where distance is assigned without a suffix
rg "(distance|length)\s*[:=]" --type ts --type tsx --type js --type jsx src/
```

For each hit, classify into one of four categories:

| Category | Example | Action |
|---|---|---|
| A. Already correctly suffixed and correct value | `const distance_km: number = 32.5` | Skip |
| B. Suffixed but wrong unit (lies) | `const distance_km = 32500` (actually meters) | Fix value or rename |
| C. Unsuffixed | `const distance = response.distance` (meters from router) | Rename to `_m` or `_km`, convert if needed |
| D. Ambiguous (need to investigate) | `const dist = compute(...)` | Read context, classify, then fix |

Output: a single markdown file `audit-report.md` listing every distance variable found, its category, file:line, current name, intended unit, and proposed action.

**Stop after Step 1 and present audit-report.md to Travis for review before proceeding to Step 2.** This is critical — silent renames can break things in ways that are hard to detect.

### Step 2: Add boundary converters

Create a single utility module if one doesn't exist:

```ts
// src/utils/distanceUnits.ts
export const M_TO_KM = (m: number): number => m / 1000;
export const KM_TO_M = (km: number): number => km * 1000;

// Runtime assertion helpers (dev-only)
export function assertKm(value: number, fieldName: string): void {
  if (process.env.NODE_ENV !== 'production') {
    // Heuristic: distances >10000 are almost certainly meters mistakenly labeled km
    if (value > 10000) {
      console.warn(`[unit-contract] ${fieldName}=${value} looks like meters, expected km`);
    }
  }
}

export function assertMeters(value: number, fieldName: string): void {
  if (process.env.NODE_ENV !== 'production') {
    // Heuristic: distances <1 with _m label are likely km
    if (value > 0 && value < 1) {
      console.warn(`[unit-contract] ${fieldName}=${value} looks like km, expected m`);
    }
  }
}
```

The 10000-km threshold catches the canonical bug (45,300 km from meters). The <1m threshold catches reverse mistakes. Adjust if you find legitimate edge cases.

### Step 3: Fix each category-C and category-D site

For each unsuffixed distance variable:

1. Determine the intended unit by reading callers and callees
2. Rename: `distance` → `distance_km` (or `_m`)
3. If the underlying value is in the wrong unit, add conversion at the assignment site using the converters
4. If the value comes from a router/API response, the conversion happens *at the boundary*: the API client wraps the raw response and exposes only suffixed fields

Example boundary fix:

```ts
// Before (suspect):
const result = await getSmartCyclingRoute(waypoints);
setRouteStats({ distance: result.distance });  // unit unclear

// After:
const result = await getSmartCyclingRoute(waypoints);
// getSmartCyclingRoute's return type now has distance_m (raw from provider)
setRouteStats({ distance_km: M_TO_KM(result.distance_m) });
```

### Step 4: Fix the Zustand store

In `routeBuilderStore.js` (or its current location), update the store's TypeScript type:

```ts
// Before
routeStats: { distance: number; elevation: number; };

// After
routeStats: { distance_km: number; elevation_gain_m: number; };
```

Update every setter, getter, and consumer to use the new names. The TypeScript compiler will surface all call sites.

### Step 5: Fix the Supabase schema

In `database/create_routes_table.sql` (or wherever the schema lives), the column `distance_km` already exists. Verify:

- All Edge Functions writing to this column use km values, not meters
- All client code reading from this column treats the value as km

If any other distance columns exist (e.g., on segments, on past rides), apply the same convention.

### Step 6: Add lint rule (optional but recommended)

Add an ESLint rule that flags any variable named exactly `distance`, `length`, `radius`, etc. without a unit suffix:

```js
// eslint-plugin-tribos / rules/distance-unit-suffix.js
module.exports = {
  create(context) {
    const SUSPECT = /^(distance|length|radius|dist|len)$/i;
    const ALLOWED_SUFFIX = /(_km|_m|Km|Meters)$/;

    return {
      VariableDeclarator(node) {
        if (node.id.type !== 'Identifier') return;
        const name = node.id.name;
        if (SUSPECT.test(name) && !ALLOWED_SUFFIX.test(name)) {
          context.report({
            node,
            message: `Distance variable "${name}" needs _km or _m suffix`,
          });
        }
      },
    };
  },
};
```

If this is too disruptive to add now, document the rule as a follow-up.

### Step 7: Add runtime assertions at high-risk sites

For each of these sites, add an `assertKm` or `assertMeters` call:

- Every `setRouteStats` call
- Every router response handler
- Every Supabase write of a distance column
- Every Supabase read of a distance column

These fire only in dev. They will catch regressions during feature work.

### Step 8: Test

Run the full existing test suite. All tests must pass with their existing assertions (no test should need to change because of a unit fix — if it does, the test was masking the bug).

Add at least these new tests:

```ts
describe('distance unit contract', () => {
  it('store distance_km accepts km values without conversion', () => { ... });
  it('router response is converted at boundary', () => { ... });
  it('assertKm warns on suspiciously large value', () => { ... });
  it('assertMeters warns on suspiciously small value', () => { ... });
});
```

### Step 9: Document

Update `CONTRIBUTING.md` or the equivalent house-rules doc with:

> **Distance unit convention:** All distance variables must end in `_km` (kilometers) or `_m` (meters). Conversions happen at module boundaries via `M_TO_KM` / `KM_TO_M`. Never name a variable just `distance`.

## Risk notes

- **Renames are the easy part; conversion fixes are the dangerous part.** A site that was previously storing meters into a "km" field will, after the fix, store the converted km value — which means *historical data may be wrong by 1000×*. Check Supabase for any rows where `distance_km > 1000` (suspicious for cycling). If any exist, flag them in the audit report and decide whether to backfill-correct.

- **The route builder store is persisted to localStorage.** Existing users have local state with the old field names. Add a migration that runs on hydration:
  ```ts
  if (state.routeStats?.distance !== undefined && state.routeStats?.distance_km === undefined) {
    // Old field present, new field missing — likely the meters-as-km bug
    state.routeStats.distance_km = state.routeStats.distance > 1000
      ? state.routeStats.distance / 1000  // was meters
      : state.routeStats.distance;        // was already km
    delete state.routeStats.distance;
  }
  ```
  Document this migration explicitly.
