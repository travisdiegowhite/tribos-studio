# T1.4 — PostHog Baseline Instrumentation

**Status:** Ready for handoff
**Priority:** High — required for measuring rebuild impact
**Estimate:** 1 day
**Depends on:** T1.3 (fallback telemetry already wired) — useful but not strictly required
**Blocks:** Nothing strictly, but should ship before T2.x begins

---

## Goal

Establish a complete PostHog event baseline for the *current* Route Builder pipeline so that the executor rebuild (T2.x) and the conversational layer (T3.x) can be measured against pre-rebuild performance.

After this task, every meaningful action in the Route Builder emits a structured event with consistent properties. The events form a funnel that can answer:

- How many users start a route generation, and how many complete it?
- How long does generation take (Claude call, routing calls, total)?
- How often do users edit a generated route vs accept it as-is?
- How often do users save? Export? Discard?
- Which routing provider succeeded? How often does fallback fire?
- Where do users abandon?

This is measurement infrastructure, not a feature. The goal is *data*, not *changes to user behavior*.

---

## Background

Per the original architectural review (§9.11):

> No usage analytics — `trackFeature` imported but never called during generate/save/export

Per memory: PostHog is already wired into Tribos (instrumentation was added in earlier phases). The issue is not that PostHog is missing — it's that Route Builder specifically lacks coverage.

Per the broader rebuild plan: without baseline data, the executor rebuild will land with no way to know whether it improved anything. This is the cheapest insurance Travis can buy before T2 starts.

## Event taxonomy

### Naming convention

All Route Builder events use the prefix `route_builder_`. Verbs in past tense. Snake_case.

Examples: `route_builder_generation_started`, `route_builder_route_saved`, `route_builder_edit_applied`.

### Common properties (every event)

Every event includes these properties:

```ts
{
  user_id: string;
  session_id: string;          // generated per browser session; persists in sessionStorage
  generation_id: string | null; // links events to a specific generation attempt
  route_id: string | null;     // present once a route exists (UUID from store or DB)
  builder_mode: "ai" | "manual" | "editing";
  source: "route_builder_page" | "training_calendar" | "coach_check_in" | "other";
  timestamp: string;           // ISO 8601 — PostHog adds its own, but this is explicit
}
```

Implement this as a helper:

```ts
// src/utils/routeBuilderTelemetry.ts
import posthog from 'posthog-js';

let cachedSessionId: string | null = null;

function getSessionId(): string {
  if (cachedSessionId) return cachedSessionId;
  cachedSessionId = sessionStorage.getItem('rb_session_id');
  if (!cachedSessionId) {
    cachedSessionId = crypto.randomUUID();
    sessionStorage.setItem('rb_session_id', cachedSessionId);
  }
  return cachedSessionId;
}

export function trackRouteBuilder(
  event: string,
  properties: Record<string, any> = {}
): void {
  posthog.capture(`route_builder_${event}`, {
    session_id: getSessionId(),
    timestamp: new Date().toISOString(),
    ...properties,
  });
}
```

Every event call site uses this helper. No raw `posthog.capture` calls in Route Builder code after this lands.

### The event catalog

These are the events to emit. Each row: event name, when fired, required properties beyond common.

| Event | When | Properties |
|---|---|---|
| `generation_started` | User clicks "Generate" in AI mode | `generation_id` (new), `training_goal`, `time_available_minutes`, `route_type`, `route_profile`, `explicit_distance_km` (nullable), `start_coord_set` (boolean), `race_type` (nullable), `race_date` (nullable) |
| `generation_context_built` | After context gathering completes, before Claude call | `generation_id`, `duration_ms`, `past_rides_count`, `familiar_segments_count`, `has_training_context` (boolean) |
| `generation_claude_called` | Just before the Claude API call | `generation_id`, `prompt_length_chars` |
| `generation_claude_responded` | When Claude returns successfully | `generation_id`, `duration_ms`, `suggestions_count`, `tokens_used` (if available) |
| `generation_claude_failed` | When Claude call fails (error, timeout, malformed) | `generation_id`, `duration_ms`, `failure_kind` ("timeout" \| "5xx" \| "429" \| "401" \| "malformed" \| "empty_response" \| "other"), `error_message` (truncated to 200 chars) |
| `generation_routing_called` | For each routing call (one per suggestion × per segment for iterative) | `generation_id`, `provider` ("stadia" \| "brouter" \| "mapbox"), `profile`, `waypoint_count` |
| `generation_routing_succeeded` | Routing call returned a route | `generation_id`, `provider`, `duration_ms` |
| `generation_routing_failed` | Routing call failed | `generation_id`, `provider`, `duration_ms`, `failure_reason` (string, truncated) |
| `generation_completed` | All suggestions built, returned to UI | `generation_id`, `total_duration_ms`, `suggestions_count`, `fallback_used` (boolean), `fallback_tier` (1/2/3, nullable) |
| `generation_failed` | Pipeline failed entirely (rare; fallback should prevent this) | `generation_id`, `total_duration_ms`, `failure_kind` |
| `generation_abandoned` | User navigates away or starts new generation before this one finishes | `generation_id`, `elapsed_ms`, `stage` ("context" \| "claude" \| "routing" \| "analysis") |
| `suggestion_selected` | User picks one of the 3 suggestions | `generation_id`, `suggestion_index` (0/1/2), `was_fallback` (boolean), `distance_km`, `elevation_gain_m` |
| `route_edit_started` | User opens AI edit panel or enters manual edit mode | `route_id`, `edit_mode` ("ai" \| "manual") |
| `route_edit_applied` | An edit successfully changed the route | `route_id`, `edit_mode`, `edit_type` (string — current taxonomy is loose; "ai_chat_message" \| "drag_waypoint" \| "add_waypoint" \| "remove_waypoint" \| "reverse" \| "snap_to_roads" \| "other"), `duration_ms`, `distance_change_km`, `elevation_change_m` |
| `route_edit_failed` | An edit failed (router error, etc.) | `route_id`, `edit_mode`, `edit_type`, `failure_reason` |
| `route_saved` | Route saved to Supabase | `route_id`, `is_new` (boolean — first save vs update), `distance_km`, `elevation_gain_m`, `generated_by` ("ai" \| "manual"), `was_fallback` (boolean), `time_from_generation_to_save_seconds` |
| `route_exported` | User exports to GPX/TCX/FIT | `route_id`, `format` ("gpx" \| "tcx" \| "fit"), `distance_km` |
| `route_discarded` | User discards a route without saving | `route_id` (or `generation_id` if no route_id yet), `had_edits` (boolean) |
| `route_opened` | User loads a previously saved route | `route_id`, `time_since_creation_days` |
| `route_fallback_used` | Already implemented in T1.3 — verify alignment | (existing properties) |
| `provider_fallback_chain_advanced` | First-choice routing provider failed, fell to next | `generation_id` (or `route_id`), `from_provider`, `to_provider`, `failure_reason` |
| `poi_layer_toggled` | User shows/hides POI overlay | `route_id`, `layer` ("cafes" \| "water" \| "rest" \| "bike_shops"), `state` ("shown" \| "hidden") |
| `elevation_profile_viewed` | User opens elevation profile | `route_id` |
| `segment_alternative_explored` | User opens segment alternatives panel for a section | `route_id`, `segment_position_km` |

### Property type discipline

- Distances always with `_km` or `_m` suffix per T1.1
- Coordinates not logged directly in events (PII risk + bulk); log only `start_coord_set: boolean` or hashed region identifiers
- Error messages truncated to 200 chars to avoid blowing up PostHog payload size
- Booleans are explicit `true`/`false`, never null
- Durations always in `_ms` integer

## Step-by-step process

### Step 1: Discovery

Run inventory of existing PostHog calls in Route Builder code:

```bash
rg "posthog\.capture|trackFeature|track\(" --type ts --type tsx --type js --type jsx src/
```

For each existing call, document in `posthog-audit.md`:
- file:line
- event name
- properties currently sent
- whether it conflicts with the catalog above (yes → flag for resolution, no → keep as-is or align property names)

The audit may surface events already emitted under different names. The principle: *keep existing event names if they have historical data*; align property naming forward. Don't break continuity.

Present `posthog-audit.md` to Travis before adding any new events.

### Step 2: Build the helper

Create `src/utils/routeBuilderTelemetry.ts` with the `trackRouteBuilder` helper per the snippet above.

Add a generation_id manager:

```ts
let currentGenerationId: string | null = null;

export function startGenerationId(): string {
  currentGenerationId = crypto.randomUUID();
  return currentGenerationId;
}

export function getCurrentGenerationId(): string | null {
  return currentGenerationId;
}

export function clearGenerationId(): void {
  currentGenerationId = null;
}
```

This is module-singleton state — fine for a single-tab UX but document the limitation. If a user opens two tabs and generates simultaneously, the generation_ids will collide. Acceptable risk for v1.

### Step 3: Wire events in priority order

Implement events in this order so partial progress still yields useful data:

**Priority 1 — generation funnel (the most important data):**
- `generation_started`
- `generation_completed`
- `generation_failed`
- `generation_abandoned`
- `suggestion_selected`

This alone tells you: how many generations succeed? How long do they take? Which suggestion gets picked?

**Priority 2 — generation internals:**
- `generation_context_built`
- `generation_claude_called`
- `generation_claude_responded`
- `generation_claude_failed`
- `generation_routing_called` / `succeeded` / `failed`

This tells you: *where* in the pipeline does time go? *Where* do failures happen?

**Priority 3 — route lifecycle:**
- `route_saved`
- `route_exported`
- `route_discarded`
- `route_opened`

This tells you: how many generations turn into saved routes? How many get exported?

**Priority 4 — editing:**
- `route_edit_started`
- `route_edit_applied`
- `route_edit_failed`

This tells you: how much do users edit generated routes? AI edit vs manual?

**Priority 5 — secondary features:**
- `provider_fallback_chain_advanced`
- `poi_layer_toggled`
- `elevation_profile_viewed`
- `segment_alternative_explored`

This tells you: which features actually get used?

Ship each priority as a separate commit within the PR for cleaner review. Or split into two PRs (P1+P2 in one, P3+P4+P5 in another) if scope feels large.

### Step 4: Abandonment detection

`generation_abandoned` is harder than the others. It needs to fire when:

- User navigates away from Route Builder during generation
- User starts a new generation while one is in flight
- User closes the tab during generation

Implementation strategy:

```ts
// Inside generateAIRoutes or equivalent
const abandonController = new AbortController();
const generationId = startGenerationId();
const stageRef = { current: 'context' as 'context' | 'claude' | 'routing' | 'analysis' };
const startTime = Date.now();

// Cleanup hook
const cleanup = () => {
  abandonController.abort();
  if (stageRef.current !== 'completed' && stageRef.current !== 'failed') {
    trackRouteBuilder('generation_abandoned', {
      generation_id: generationId,
      elapsed_ms: Date.now() - startTime,
      stage: stageRef.current,
    });
  }
};

// Register cleanup on unmount
useEffect(() => {
  return cleanup;
}, []);

// Register cleanup on page hide
useEffect(() => {
  const onHide = () => {
    if (document.visibilityState === 'hidden') {
      cleanup();
    }
  };
  document.addEventListener('visibilitychange', onHide);
  return () => document.removeEventListener('visibilitychange', onHide);
}, []);
```

Update `stageRef.current` at each pipeline stage transition. Mark `completed` or `failed` on terminal state so the abandon hook becomes a no-op.

**Important:** PostHog events fired on `visibilitychange: hidden` should use `posthog.capture(..., { send_instantly: true })` or the equivalent flag for the installed version, so they fire before the tab actually closes.

### Step 5: Routing provider attribution

The current `smartCyclingRouter.js` has a fallback chain (BRouter → Stadia → Mapbox or the reverse). The instrumentation needs to capture *which* provider actually returned the route, not just that routing succeeded.

In `smartCyclingRouter.js` (and any equivalent), every provider attempt emits `generation_routing_called` on enter and `generation_routing_succeeded` or `generation_routing_failed` on result. When falling through, emit `provider_fallback_chain_advanced`.

This is the most invasive piece of the work — multiple sites in `smartCyclingRouter.js` need wiring. Audit the file first; map every `try { stadia... } catch { brouter... }` style block.

### Step 6: Save/export wiring

In `routesService.saveRoute` (or wherever the save call lives), emit `route_saved` on success. Include `time_from_generation_to_save_seconds` computed from the generation_id's start timestamp (need to stash that somewhere — Zustand store works).

Same pattern for export: every export path (GPX, TCX, FIT) emits `route_exported` with the format.

### Step 7: Edit wiring

`route_edit_started` fires when the AI Edit Panel opens OR when manual edit mode is entered. `route_edit_applied` fires when an edit actually changes route state. Distinguish:

- AI edit: each user message in the AI chat that produces a route change → `edit_type: "ai_chat_message"`
- Manual edits: each waypoint operation → `edit_type` per the action

`route_edit_failed` fires when an edit attempt errors (router unavailable, intent unclear, etc.).

### Step 8: Validation

Before merging, generate test events end-to-end:

1. Open Route Builder
2. Generate a route — verify `generation_started`, `generation_context_built`, `generation_claude_called`, `generation_claude_responded`, `generation_routing_called` × N, `generation_routing_succeeded` × N, `generation_completed` all fire
3. Select a suggestion — verify `suggestion_selected`
4. Edit the route via AI chat — verify `route_edit_started`, `route_edit_applied`
5. Drag a waypoint — verify `route_edit_applied` with manual `edit_type`
6. Save — verify `route_saved`
7. Export GPX — verify `route_exported`
8. Open Route Builder fresh; generate; navigate away mid-generation — verify `generation_abandoned`
9. Disable Claude temporarily; generate; verify `generation_claude_failed` and `route_fallback_used` (from T1.3) both fire

Each verification: check the PostHog live event stream and confirm event + properties match.

### Step 9: Document the catalog

Create `docs/route-builder-telemetry.md` (or equivalent) with the full event catalog above. This becomes the reference for anyone (you, future Claude Code agents, anyone Travis brings on) wiring or reading these events.

The catalog should be a markdown table identical in structure to the table in this spec, kept in the repo. When the executor lands, T2.x events get added to that doc.

### Step 10: Build a baseline data freeze

After events have been flowing for ~1 week of normal use (or whatever's available pre-rebuild), produce a snapshot:

- Median and p95 `generation_completed.total_duration_ms`
- Generation success rate (`generation_completed` / `generation_started`)
- Fallback rate (`fallback_used: true` / `generation_completed`)
- Edit rate (`route_edit_applied` users / `suggestion_selected` users)
- Save rate (`route_saved` / `suggestion_selected`)
- Provider win rate (`generation_routing_succeeded` by provider)

Store this in `docs/route-builder-baseline-YYYY-MM-DD.md`. This is the number to beat when T2 ships.

## Risk notes

- **PostHog rate limits.** High-frequency events (especially `generation_routing_called` during iterative routing) can fire dozens of times per generation. PostHog's free tier has event ingestion limits. Check current usage; if approaching limits, consider sampling routing-call events (e.g., emit every 5th call with a `sample_rate` property) rather than every one. This is a deferrable decision — ship without sampling first, add it if costs bite.

- **Event property drift.** As code changes, properties on events drift if not enforced. Consider a lightweight runtime validator on `trackRouteBuilder` that warns if a known event is called without expected properties. Optional for v1.

- **The abandonment event is the trickiest.** It must fire reliably on tab close, which means it needs `sendBeacon` or the equivalent PostHog instant-send flag. Test this specifically — it's easy to ship and silently never fire.

- **Don't over-instrument.** The temptation is to track everything "just in case." Resist. The catalog above covers the funnel that matters. Adding 20 more events makes the data harder to use, not easier.

- **PII discipline.** Coordinates aren't in the catalog by design. If a future event needs geographic data, use bounding-box quantization or hashed region IDs. Never log raw user-route coordinates to PostHog. The user can be re-identified from route geometry.

- **The baseline freeze (Step 10) is non-negotiable.** Without it, the rebuild lands and you have no comparison. One week of data is the minimum; two is better. Travis should not start T2.1 implementation until baseline numbers are captured.
