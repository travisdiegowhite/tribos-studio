# T1.3 — Claude-Failure Heuristic Fallback

**Status:** Ready for handoff
**Priority:** Critical — user-facing reliability
**Estimate:** 2–3 days
**Depends on:** Nothing (can run concurrent with T1.1 and T1.2)
**Blocks:** Beta launch readiness

---

## Goal

Eliminate the "Claude failed → empty array → user sees nothing" failure path in the current Route Builder. When the LLM call fails for any reason, the system produces a usable route via deterministic heuristic generation, with clear UI signaling that this is a fallback (not a failure).

This is the most user-facing of the three foundation tasks. It changes the worst-case UX from "broken" to "degraded but useful."

---

## Background

Per the Route Builder architectural review:

> **No fallback if Claude fails** — returns `[]`, user sees nothing.

And per the broader conversation: in a conversational route builder, *silence is the worst possible response*. A wrong route is recoverable ("no, more like this"). An empty result is a dead end.

This task implements a heuristic fallback that produces a real route when Claude is unavailable, hitting rate limits, returning malformed JSON, or just being slow.

## The fallback strategy

When Claude fails, the fallback generates **one route** (not three) using these heuristics, in priority order:

### Tier 1 — Familiar-loop fallback (preferred)

If user has Strava history with rides in the start region:

1. Query past rides whose geometry intersects the start coordinate bounding box (configurable radius, default 25km)
2. Filter to rides matching the requested training goal (endurance, tempo, etc.) — fall back to "any goal" if too few
3. Pick a ride whose distance is within ±25% of the target distance
4. Use that ride's geometry as the starting point
5. If the ride's start point is different from the user's requested start, prepend a short connector segment using the standard smart cycling router

This gives the user "a route you've ridden before" instead of nothing. Familiar = safe.

### Tier 2 — Radial-loop fallback

If no familiar rides match:

1. Compute target distance from requested time and user's speed profile
2. Generate a simple radial-loop route:
   - From start, head in cardinal direction (rotate quarterly: N, E, S, W based on day-of-week to vary)
   - Travel target_distance / 4 in that direction
   - Turn 90° (right) and travel target_distance / 4
   - Return to start via remaining waypoints
3. Snap-to-roads via smart cycling router with the user's profile (road/gravel/mtb)

This produces a generic but valid route. Not personalized, but functional.

### Tier 3 — Last-resort minimal route

If even Tier 2 fails (router unavailable):

1. Generate a straight out-and-back from start
2. Length = target_distance / 2 in arbitrary direction
3. Return error indication to UI: "Tried our best — couldn't reach the routing service. Try again in a moment."

This last tier should be exceedingly rare in practice — it fires only if both Claude AND the smart cycling router are down simultaneously.

## Step-by-step process

### Step 1: Inventory current failure paths

Find every place in the current code where Claude can fail. Grep for:

```bash
rg "claudeRouteService|claude-routes|generateAIRoutes" --type ts --type js src/
```

For each call site, identify:
- What happens on timeout?
- What happens on 429?
- What happens on 401?
- What happens on 5xx?
- What happens on malformed JSON response?
- What does the caller see?

Document this as `claude-failure-modes.md`. Each row: file:line, failure trigger, current behavior, target behavior (delegate to fallback).

### Step 2: Build the fallback module

Create `src/utils/routeGenerationFallback.ts`:

```ts
import type { Coordinate } from '../types/geo';
// Note: depends on T1.2 (Coordinate type). If T1.2 hasn't shipped yet,
// use whatever the current coordinate type is and document the dependency.

type FallbackInput = {
  startCoord: Coordinate;
  targetDistanceKm: number;
  trainingGoal: TrainingGoal;
  routeProfile: RoutingProfile;
  userId: string;
};

type FallbackResult = {
  geometry: Coordinate[];
  waypoints: Waypoint[];
  stats: { distance_km: number; elevation_gain_m: number };
  fallback_tier: 1 | 2 | 3;
  fallback_reason: string;
};

export async function generateFallbackRoute(input: FallbackInput): Promise<FallbackResult> {
  // Try Tier 1
  try {
    const result = await tryFamiliarLoop(input);
    if (result) return { ...result, fallback_tier: 1, fallback_reason: 'claude_unavailable' };
  } catch (e) {
    console.warn('[fallback] Tier 1 (familiar loop) failed:', e);
  }

  // Try Tier 2
  try {
    const result = await tryRadialLoop(input);
    if (result) return { ...result, fallback_tier: 2, fallback_reason: 'claude_unavailable' };
  } catch (e) {
    console.warn('[fallback] Tier 2 (radial loop) failed:', e);
  }

  // Tier 3 — always succeeds
  return generateOutAndBack(input);
}
```

Implementation notes:

- `tryFamiliarLoop`: queries Supabase via existing route familiarity infrastructure (`/api/road-segments` if usable, or direct query). Returns `null` if no candidates found.
- `tryRadialLoop`: uses existing smart cycling router. The rotation-by-day-of-week is a deliberate variety injection so users don't get the same route twice.
- `generateOutAndBack`: pure geometry, no router call. Can't fail.

### Step 3: Integrate into `aiRouteGenerator`

In `generateAIRoutes()`, find the Claude call site (around line 145–150 per the architectural review). Wrap it:

```ts
async function generateAIRoutes(input) {
  // ...existing context gathering...

  let aiResult: AIResult | null = null;
  let fallbackUsed = false;

  try {
    aiResult = await claudeRouteService.generate(prompt, options);
  } catch (e) {
    console.warn('[generateAIRoutes] Claude failed, falling back:', e);
    fallbackUsed = true;
  }

  if (!aiResult || aiResult.suggestions.length === 0) {
    fallbackUsed = true;
  }

  if (fallbackUsed) {
    const fallback = await generateFallbackRoute({
      startCoord: input.startCoord,
      targetDistanceKm: input.targetDistanceKm,
      trainingGoal: input.trainingGoal,
      routeProfile: input.routeProfile,
      userId: input.userId,
    });

    return {
      suggestions: [{
        ...fallback,
        isFallback: true,
        fallbackTier: fallback.fallback_tier,
      }],
    };
  }

  // ...existing post-Claude pipeline...
}
```

Key behaviors:

- Fallback fires on *both* exception and empty-suggestions result. The current "returns `[]`" bug is just as bad as a thrown error.
- Fallback returns ONE route, not three. The UI handles this via the `isFallback` flag.
- The original aiResult flow is untouched on the success path.

### Step 4: UI signaling

When a fallback route is shown, the user must know. Not in a way that says "this is broken," but in a way that says "AI was unavailable, here's something useful."

Update the route suggestion display:

```tsx
{route.isFallback && (
  <div className="route-fallback-banner" /* style: subtle, not alarming */>
    {route.fallbackTier === 1 && (
      <span>Based on a route you've ridden before — AI assist will be back soon.</span>
    )}
    {route.fallbackTier === 2 && (
      <span>Generic route while AI is unavailable. Try again for a personalized version.</span>
    )}
    {route.fallbackTier === 3 && (
      <span>Limited connectivity — basic out-and-back generated. Try again in a moment.</span>
    )}
  </div>
)}
```

Visual treatment per Tribos brand: use the gold accent (`#C49A0A`) for Tier 1, coral (`#C43C2A`) for Tier 3. Tier 2 in neutral text. Banner is small, dismissable, never modal.

### Step 5: Telemetry

Every fallback occurrence emits a PostHog event:

```ts
posthog.capture('route_fallback_used', {
  tier: fallback.fallback_tier,
  reason: fallback.fallback_reason,
  user_id: input.userId,
  training_goal: input.trainingGoal,
  target_distance_km: input.targetDistanceKm,
});
```

This is essential. You need to know:
- How often fallbacks fire (rate)
- Which tier (severity)
- What the trigger was (Claude timeout vs 429 vs malformed JSON)
- Whether specific users hit it disproportionately

### Step 6: Add timeout to the Claude call

The current pipeline has no timeout on Claude. Add one:

```ts
const CLAUDE_TIMEOUT_MS = 15000; // 15s

async function callClaudeWithTimeout(prompt, options) {
  const controller = new AbortController();
  const timeout = setTimeout(() => controller.abort(), CLAUDE_TIMEOUT_MS);
  try {
    return await fetch('/api/claude-routes', {
      method: 'POST',
      body: JSON.stringify({ prompt, options }),
      signal: controller.signal,
    });
  } finally {
    clearTimeout(timeout);
  }
}
```

Timeouts → caught by Step 3 try/catch → fallback fires. The 15s value is per the Turn Model Spec §12. Adjust if real-world Claude latency varies.

## Risk notes

- **Familiar-loop fallback depends on Strava history.** New users with no rides will always fall through to Tier 2. That's correct — Tier 2 is still better than nothing. Don't try to be clever about new users; the radial loop is fine for them.

- **The radial-loop direction-by-day-of-week is a heuristic.** It exists so users don't get the same fallback route twice in a row. If you find a better variety strategy (e.g., direction-by-last-fallback-direction), use it. The point is *not the same route every time*.

- **Tier 3 is a true last-resort.** If it fires more than ~0.1% of attempts, something is structurally wrong (probably the smart cycling router). Watch the PostHog telemetry; if Tier 3 rate climbs, that's a signal to investigate routing infrastructure, not the fallback code.

- **The fallback banner must not feel like an error.** Tier 1 is "AI is taking a break, here's something familiar" — actually a *better* experience than the average AI suggestion for some users. Treat it accordingly in copy and visual weight.

- **Don't catch errors silently anywhere.** Every catch site logs and emits telemetry. The whole point is to know when this is firing.
