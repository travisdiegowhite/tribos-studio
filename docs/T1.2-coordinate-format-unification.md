# T1.2 — Coordinate Format Unification

**Status:** Ready for handoff
**Priority:** Critical — blocks T2 (executor rebuild)
**Estimate:** 1–2 days
**Depends on:** Nothing (can run concurrent with T1.1)
**Blocks:** T2.x (all executor work)

---

## Goal

Standardize all internal coordinate representations to `[lng, lat]` arrays (GeoJSON convention). After this task, conversion to/from other formats (`{lat, lng}`, `{lat, lng, elevation}`) happens only at explicit module boundaries, each of which is documented.

This is the precondition for the new executor architecture, which mandates `[lng, lat]` everywhere internally.

---

## Background

Per the Route Builder architectural review:

> **Three coordinate formats in flight:** `[lng,lat]`, `{lat,lng}`, `{lat,lng,elevation}` — easy to swap.

Coordinate format mismatches are a particularly nasty bug class because the values are both numbers in similar ranges (lat: -90 to 90, lng: -180 to 180). A swap doesn't crash — it just renders the route on the wrong continent. In Colorado, lat ≈ 40 and lng ≈ -105, so a swap puts you in Antarctica.

---

## Scope

### In scope

- All TypeScript/JavaScript files under `src/` that handle coordinates
- Type definitions for routes, waypoints, POIs, segments
- Mapbox GL integration (uses `[lng, lat]` natively — good)
- Stadia Maps router (Valhalla — uses `{lat, lon}`)
- BRouter (uses `lon,lat` query strings)
- Open-Elevation API (uses `{latitude, longitude}`)
- Supabase storage formats for geometry, waypoints

### Out of scope

- Other geographic types (BBox, viewport) — separate audit if needed; they're less error-prone
- UI display strings — those are formatted at presentation
- Mapbox tile rendering — Mapbox handles its own format internally

---

## Definitions

| Term | Meaning |
|---|---|
| Canonical format | `[lng, lat]` — tuple of numbers, longitude first |
| External format | Any other shape: `{lat, lng}`, `{lat, lon}`, `{latitude, longitude}`, `"lng,lat"` strings, etc. |
| Boundary | Code site where coordinates enter from or exit to a third-party API |

## Step-by-step process

### Step 1: Discovery

Produce an audit report listing every coordinate-handling site. Use ripgrep:

```bash
# Find coordinate-shaped object patterns
rg "\{\s*lat\s*:" --type ts --type tsx --type js --type jsx src/
rg "\{\s*latitude\s*:" --type ts --type tsx --type js --type jsx src/
rg "\{\s*lng\s*:" --type ts --type tsx --type js --type jsx src/
rg "\{\s*lon\s*:" --type ts --type tsx --type js --type jsx src/

# Find array coordinate patterns
rg "\[\s*-?\d+(\.\d+)?\s*,\s*-?\d+(\.\d+)?\s*\]" --type ts --type tsx src/

# Find type definitions
rg "Coordinate|LatLng|LngLat|GeoPoint" --type ts --type tsx src/
```

For each hit, classify:

| Category | Example | Action |
|---|---|---|
| A. Already `[lng, lat]` internal | `const coord: [number, number] = [-105, 40]` | Skip, but verify it's typed |
| B. External format, used internally | `const point = { lat: 40, lng: -105 }; routeBuilder.addPoint(point)` | Convert at boundary; use canonical internally |
| C. External format at a real boundary | `mapboxApi.directions({ coordinates: [{ lat, lng }] })` | Wrap in a documented converter |
| D. Reversed by accident | `[40, -105]` where `[lng, lat]` expected | Fix — high-priority bug |
| E. Mixed in same module | Some functions use array, others use object | Standardize to canonical |

Output: `coord-audit-report.md` listing each site, file:line, current format, target format, and proposed action.

**Stop after Step 1 and present audit-report to Travis before proceeding.** Coordinate fixes are subtle; review them before touching code.

### Step 2: Define the canonical type

Create or update a shared types module:

```ts
// src/types/geo.ts

/**
 * Canonical coordinate format: [longitude, latitude].
 * GeoJSON convention. Used internally throughout the codebase.
 *
 * For longitude: -180 to 180 (negative = western hemisphere)
 * For latitude:  -90 to 90  (negative = southern hemisphere)
 *
 * IMPORTANT: This is the ONLY coordinate type allowed in internal code.
 * Conversion to/from other formats happens at module boundaries.
 */
export type Coordinate = readonly [longitude: number, latitude: number];

/**
 * Helper to construct a Coordinate from longitude/latitude.
 * Use this whenever you need to be explicit about ordering.
 */
export const coord = (lng: number, lat: number): Coordinate => [lng, lat];

/**
 * Validate a value is a plausible Coordinate.
 * Returns true if shape and value ranges are valid.
 */
export function isValidCoordinate(value: unknown): value is Coordinate {
  if (!Array.isArray(value) || value.length !== 2) return false;
  const [lng, lat] = value;
  if (typeof lng !== 'number' || typeof lat !== 'number') return false;
  if (lng < -180 || lng > 180) return false;
  if (lat < -90 || lat > 90) return false;
  return true;
}

/**
 * Dev-only assertion. No-op in production.
 * Logs a warning if a coordinate looks reversed for the expected region.
 */
export function assertCoordinate(value: unknown, fieldName: string): void {
  if (process.env.NODE_ENV === 'production') return;
  if (!isValidCoordinate(value)) {
    console.warn(`[coord-contract] ${fieldName} is not a valid [lng, lat]:`, value);
    return;
  }
  // Heuristic check: in Tribos's primary region (US), lng is negative and lat is positive.
  // A reversed coord would look like [positive lng, negative lat].
  const [lng, lat] = value;
  if (lng > 0 && lat < 0 && Math.abs(lng) < 90 && Math.abs(lat) < 90) {
    console.warn(
      `[coord-contract] ${fieldName}=${JSON.stringify(value)} looks reversed (lat,lng instead of lng,lat)`
    );
  }
}
```

### Step 3: Create explicit boundary converters

For each third-party API that uses a non-canonical format, create a documented converter:

```ts
// src/utils/coordConverters.ts

import type { Coordinate } from '../types/geo';

// ---- Mapbox ----
// Mapbox GL JS uses [lng, lat] natively (matches canonical). No converter needed.
// Mapbox Directions API: also uses [lng, lat]. No converter needed.

// ---- Stadia Maps (Valhalla) ----
// Stadia uses { lat, lon } in request and response bodies.
export function stadiaToCanonical(p: { lat: number; lon: number }): Coordinate {
  return [p.lon, p.lat];
}
export function canonicalToStadia(c: Coordinate): { lat: number; lon: number } {
  return { lon: c[0], lat: c[1] };
}

// ---- BRouter ----
// BRouter takes "lonlats" as semicolon-separated "lon,lat" pairs in query strings.
export function canonicalToBRouter(coords: Coordinate[]): string {
  return coords.map(([lng, lat]) => `${lng},${lat}`).join('|');
}
export function brouterToCanonical(geometry: number[][]): Coordinate[] {
  // BRouter returns [lng, lat, elevation] tuples — drop elevation.
  return geometry.map(([lng, lat]) => [lng, lat] as Coordinate);
}

// ---- Open-Elevation ----
// Uses { latitude, longitude }.
export function canonicalToOpenElevation(c: Coordinate): { latitude: number; longitude: number } {
  return { latitude: c[1], longitude: c[0] };
}

// ---- Garmin/Strava activity geometry ----
// Both use [lat, lng] in their FIT/GPX exports. Document and convert.
export function activityToCanonical(p: [number, number]): Coordinate {
  return [p[1], p[0]]; // swap
}
```

Every boundary gets its own named converter. No inline conversions allowed.

### Step 4: Replace internal usage

For each Category B/D/E site from the audit, replace the external format with canonical:

```ts
// Before
function addWaypoint(waypoint: { lat: number; lng: number }) {
  setWaypoints([...waypoints, waypoint]);
}

// After
function addWaypoint(waypoint: Coordinate) {
  setWaypoints([...waypoints, waypoint]);
}
```

Update all callers. The TypeScript compiler is your friend here — let it surface every site that needs to change.

### Step 5: Update type definitions

Find every type/interface that holds a coordinate and update:

```ts
// Before
interface Waypoint {
  lat: number;
  lng: number;
  name?: string;
}

// After
interface Waypoint {
  coordinate: Coordinate;
  name?: string;
}
```

This is a breaking change for stored data. Add migrations (see Step 8).

### Step 6: Replace router/API call sites

For each call to Stadia, BRouter, Open-Elevation, etc., use the boundary converters:

```ts
// Before
const url = `${BROUTER_URL}/brouter?lonlats=${coords.map(c => `${c.lng},${c.lat}`).join('|')}`;

// After
import { canonicalToBRouter } from '../utils/coordConverters';
const url = `${BROUTER_URL}/brouter?lonlats=${canonicalToBRouter(coords)}`;
```

### Step 7: Add runtime assertions

At each boundary and at high-risk internal sites, call `assertCoordinate`:

```ts
function setRouteGeometry(geometry: Coordinate[]) {
  if (process.env.NODE_ENV !== 'production') {
    geometry.forEach((c, i) => assertCoordinate(c, `geometry[${i}]`));
  }
  // ...
}
```

### Step 8: Migrate persisted data

The route builder store and Supabase both contain coordinate data in some format. Two migrations are needed:

**Migration 8a — localStorage (Zustand store)**

In the store's `merge` or hydration function, detect old-format waypoints and convert:

```ts
function migrateWaypoint(wp: any): Waypoint {
  if (wp.coordinate && Array.isArray(wp.coordinate)) {
    return wp; // already canonical
  }
  if (typeof wp.lat === 'number' && typeof wp.lng === 'number') {
    return { coordinate: [wp.lng, wp.lat], name: wp.name };
  }
  if (typeof wp.lat === 'number' && typeof wp.lon === 'number') {
    return { coordinate: [wp.lon, wp.lat], name: wp.name };
  }
  console.warn('[migrate] Unknown waypoint shape, dropping:', wp);
  return null as any;
}
```

**Migration 8b — Supabase**

Inspect the `routes.geometry` and `routes.waypoints` JSONB columns. Run a one-time SQL migration script that:

1. SELECTs all rows
2. For each, parses the JSON and detects format
3. Converts to canonical `[lng, lat]`
4. UPDATEs the row

**This is destructive.** Run on a backup first. Verify the migration script with a few hand-picked rows before applying broadly. Get Travis to confirm before running on production data.

## Risk notes

- **Latitude/longitude swaps are nearly undetectable in code review.** Both are floats in similar ranges. The runtime assertion's heuristic catches the obvious Colorado case but won't catch all swaps. *Visual verification is required*: after the change, open the app, build a route, and confirm it appears in the correct geographic location.

- **Mapbox GL is `[lng, lat]` natively.** This is a stroke of luck — most map-rendering code is already canonical. The bugs are in router calls and stored data.

- **Stored geometry in Supabase is the riskiest piece.** Users have existing routes. If the migration script reverses an already-canonical row, those routes break. The script needs heuristic detection — see Step 8b — and a dry-run mode that reports what *would* change before applying.

- **Activity import is a hidden landmine.** FIT files and Strava polyline-decoded geometry use `[lat, lng]`. There are likely several import paths. The `activityToCanonical` converter must be applied at every one. Audit `activityRouteAnalyzer.ts` and any FIT/GPX import code specifically.
